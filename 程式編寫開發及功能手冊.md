# LINE Bot Assistant - 台灣三星電腦螢幕專屬客服

## 📋 專案資訊

「天條」非常簡單清晰：
用戶發問
├─ 精確型號 (S27CG552EC) → RULES/QA 回答 → 結束
├─ 系列別稱 (G8) → 列出 1,2,3... → 等選擇 → 鎖定型號
│ └─ 若需操作步驟 → PDF
└─ 追問 ("3" / "它怎麼開") → 用記憶中的型號繼續

LINE Bot 回答邏輯完整架構 (v29.3.9)

1. 訊息入口 (doPost → handleMessage)
   編碼 邏輯描述
   1.1 接收 LINE Webhook 事件 (doPost)
   1.2 解析 event.message.text 取得用戶訊息 msg
   1.3 識別 userId 和 contextId (群組/個人)
   1.4 重複事件過濾：檢查 webhookEventId 是否已處理過
2. 訊息分流 (handleMessage)
   編碼 條件 處理方式
   2.1 msg.startsWith("/") 或 msg === "不滿意這回答請繼續擴大搜尋" 進入 指令模式 (handleCommand)
   2.2 存在 ENTRY_DRAFT 且非指令 進入 建檔模式 (handleDraftModification)
   2.3 其他一般訊息 進入 對話模式 (主流程)
   2.4 若為新問題 (非指令) 重置「不滿意」計數器 (dissatisfied_count)
3. 指令模式 (handleCommand)
   編碼 指令 功能
   3.1 /重啟, /reboot 清除對話記憶 + 同步知識庫
   3.2 /help, /? 顯示說明文字
   3.3 /p1/p9 或 /1/9 切換 Prompt 人格
   3.4 /狀態 顯示系統健康狀態
   3.5 /記錄 進入 QA 建檔模式
   3.6 /清除 清除對話記憶 (不同步)
   3.7 不滿意這回答請繼續擴大搜尋, /擴大搜尋, /不滿意 進入 擴大搜尋流程 (見 §6)
   3.8 /重試 對上次問題強制網搜
4. 對話模式 - 主流程
   編碼 步驟 說明
   4.1 直通車關鍵字偵測 若 msg 包含 CLASS_RULES 中的別稱 (G6, M7...)，注入對應型號至 Context
   4.2 功能問題判斷 若問題涉及功能 (如「氛圍燈」) 且命中多型號 → 走 QA/RULES 優先路徑
   4.3 動態 Context 注入 (buildDynamicContext) ① QA 匹配 ② CLASS_RULES 匹配 ③ 規格反查 (CORESYNC 等)
   4.4 歷史對話載入 從 Cache 或 Sheet 讀取最近 N 輪對話
   4.5 呼叫 LLM (callLLMWithRetry) 送出 Prompt + Context + History → 取得 AI 回覆
   4.6 回覆後處理 ① Guard 糾錯 ② 格式化 ③ 附加 Quick Reply 按鈕
   4.7 歷史更新 將本輪 User+AI 訊息寫入 Cache/Sheet
5. 知識來源優先順序 (鐵律)
   優先級 來源 說明
   5.1 QA.csv 精選問答庫 (最高優先)
   5.2 CLASS_RULES.csv 型號規格表 + 系列別稱
   5.3 PDF 手冊 深度查詢 (條件觸發)
   5.4 網路搜尋 (google_search) 最終備援或用戶強制觸發
6. 擴大搜尋流程 (Smart Contrastive Search)
   編碼 步驟 說明
   6.1 讀取 dissatisfied_count (Cache) 計算本問題的重試次數
   6.2 三振出局 若 count > 3，隨機回覆「建議問 Sam」的委婉訊息並結束
   6.3 第 1 次嘗試 正常擴大搜尋：補充先前未提及的細節
   6.4 第 2~3 次嘗試 差異化搜尋：假設上一輪答案錯誤，專找「例外」「規格差異」「常見誤解」
   6.5 呼叫 callLLMWithRetry (forceWebSearch=true) 強制啟用 google_search tool
   6.6 歷史更新 將用戶指令 + AI 搜尋結果成對寫入 History
   6.7 重貼 Quick Reply 按鈕 若 count < 3，繼續顯示「不滿意這回答請繼續擴大搜尋」按鈕
7. Quick Reply 按鈕邏輯
   編碼 條件 顯示按鈕
   7.1 AI 回答完成 (非列表選擇) 「不滿意這回答請繼續擴大搜尋」
   7.2 命中多型號需選擇 1. S27xxx, 2. S32xxx... (數字+完整型號)
   7.3 擴大搜尋結果 (count < 3) 「不滿意這回答請繼續擴大搜尋」
   7.4 擴大搜尋結果 (count >= 3) 無按鈕 (三振出局)
8. LLM 呼叫邏輯 (callLLMWithRetry)
   編碼 參數 說明
   8.1 messages 含 System Prompt + History + User Query
   8.2 imageBlob 若為圖片訊息，附帶 Base64
   8.3 attachPDFs 是否載入 PDF 手冊 (Deep Mode)
   8.4 isRetry 是否為重試模式 (影響模型選擇)
   8.5 forceWebSearch 是否強制帶入 google_search tool
   8.6 模型選擇 Fast Mode: gemini-2.0-flash, Think Mode: gemini-2.0-flash
   8.7 **輸入清洗** (v29.3.47): 自動檢測並轉換非文字訊息 (如 Flex Message 物件) 為字串，防止 API 400 錯誤
9. 全域變數與設定
   編碼 變數 用途
   9.1 GAS_VERSION 當前版本號 (v29.3.9)
   9.2 ENABLE_QUICK_REPLY Quick Reply 功能開關 (true)
   9.3 LLM_PROVIDER LLM 服務提供者 (Gemini / OpenRouter)
   9.4 quickReplyOptions 全域 Quick Reply 陣列 (跨函數共享)
   9.5 CONFIG.HISTORY_PAIR_LIMIT 歷史對話保留上限 (10 對)
   9.6 CONFIG.SUMMARY_THRESHOLD 超長對話觸發摘要的門檻 (12 對)

## 🆕 v29.4.x 修復紀錄

### v29.4.0 (2026-01-15) - 二段式 AI 架構 (Smart Router)

**核心改動**：

大幅簡化 RAG 邏輯，實現「二段式 AI」架構：

1. **第一階段 (極速模式)**：直接注入 QA 全文 + 輕量層全文，讓 AI 判斷能否回答。
2. **型號選擇泡泡**：若 AI 無法回答，回覆 `[AUTO_SEARCH_PDF]` 並附上 `[型號:xxx,yyy]` 標籤，系統自動生成 Quick Reply 泡泡讓用戶選擇。
3. **第二階段 (PDF 模式)**：用戶選定型號後，載入對應 PDF 手冊回答。

**主要修改**：

- **`buildDynamicContext`**：移除分詞篩選邏輯 (~350 行)，改為直接注入 QA + 輕量層全文。
- **`Prompt.csv`**：新增 `[型號:xxx,yyy]` 標籤指令，讓 AI 在無法回答時提供型號清單。
- **暗號處理邏輯**：新增型號標籤解析與泡泡選項生成功能。
- **版本更新**：`GAS_VERSION` → `v29.4.0`，`Prompt.csv` → `v29.4.0`。

**修復問題**：

- 「40 吋」等模糊查詢無法被識別的問題 (數字被分詞丟失)。
- 程式碼過度複雜、過度篩選導致 AI 無法看到完整資訊的問題。

### v29.4.1 (2026-01-15) - 規格層注入修復

**問題**：用戶詢問「40 吋」時 AI 無法識別型號，因為 40 吋型號位於規格層 (L47+)，而 v29.4.0 只注入輕量層。

// 關鍵修復！更新對話歷史
const askMsgObj = { role: "assistant", content: askMsg };

## 🆕 v24.4 新功能：PDF 智慧匹配 + 型號反問

### 問題背景

用戶問「M8 的視訊鏡頭該如何使用」時：

- 直通車提取內部代號 `M80D`
- 但 PDF 檔名是 `S32BM801.pdf`、`S32FM803.pdf`、`S32DM803.pdf`
- `M80D` 無法匹配任何 PDF → 載入錯誤的手冊

### 解決方案

#### 1. CLASS_RULES 定義型號模式

```csv
別稱_M8,Smart Monitor M8，高階智慧螢幕，附磁吸視訊鏡頭...型號模式為：M80D或S32?M80*
```

#### 2. 新函數 `searchPdfByAliasPattern()`

從「型號模式為：XXX」提取模式，轉為正則匹配 PDF：

- `?` → `.` (任意單字元)
- `*` → `.*` (任意多字元)

#### 3. 完整流程

```
用戶：「M8 視訊鏡頭怎麼用」
  ↓
直通車命中 M8 → 記錄關鍵字（不立即開 PDF Mode）
  ↓
Fast Mode（QA + CLASS_RULES）
  ↓
AI 判斷資料不足 → 輸出 [AUTO_SEARCH_PDF]
  ↓
searchPdfByAliasPattern("M8") → 找到 3 個匹配
  ↓
反問：「M8 有幾個版本，請問型號開頭是？
  1️⃣ S32BM8...
  2️⃣ S32DM8...
  3️⃣ S32FM8...」
  ↓
用戶回覆「2」
  ↓
載入 S32DM803.pdf → 用原始問題重新回答
```

### 新增函數

| 函數                             | 用途                              |
| -------------------------------- | --------------------------------- |
| `searchPdfByAliasPattern()`      | 從 CLASS_RULES 提取模式並搜尋 PDF |
| `handlePdfSelectionReply()`      | 處理用戶的型號選擇回覆            |
| `buildPdfSelectionMessage()`     | 生成反問訊息                      |
| `checkDirectDeepSearchWithKey()` | 直通車檢查並返回命中關鍵字        |

## 🔒 安全機制

1. **事件去重** - 60 秒內相同 eventId 不重複處理
2. **Script Lock** - 同步與寫入使用鎖定機制
3. **API 重試** - 最多 3 次指數退避重試
4. **Token 過大保護** - 自動偵測並回傳錯誤訊息
5. **知識庫自動修復** - 404 錯誤時自動清除快取重建
